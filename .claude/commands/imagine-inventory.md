# Imagine Inventory: $ARGUMENTS

You are an architecture imagination agent. The user describes what they want their app to do. You synthesise a complete inventory of every file, model, service, view, block type, and component that would need to exist — even though none of it is built yet. This is a forward-looking TODO inventory, not a scan of existing code.

## Step 1: Gather context

Read these files to understand what already exists:
1. `rork.json` — project name, framework, path
2. `CLAUDE.md` — project rules, conventions, architecture patterns
3. Glob `screens/*/inventory.md` — existing screen inventories (to avoid duplicating what's already tracked)
4. Glob `ios/**/*.swift` — current file list (to know what's built vs what's needed)

Set `PROJECT_NAME` from rork.json (lowercase, hyphenated).

## Step 2: Understand the user's vision

The user wants:
> $ARGUMENTS

Parse this into:
- **Features** — distinct capabilities described (e.g. "AI chat", "speech input", "structured blocks")
- **Data types** — models, enums, structs implied
- **Services** — API integrations, file handling, audio processing
- **UI components** — views, blocks, renderers implied
- **Infrastructure** — protocols, utilities, config needed to support it all
- **Cross-platform considerations** — if mentioned, note what must be portable

## Step 3: Imagine the complete inventory

For every feature, think through the FULL chain: user interaction → view → state → service → API → response → rendering. Don't just list the obvious pieces — include:
- Protocols/interfaces that make it swappable
- Error types and error handling views
- Loading/streaming states
- Config and environment requirements
- File structure and naming conventions following CLAUDE.md rules

Classify every imagined item into sub-types:
- **model** — structs, enums, data types (mark `nonisolated` if pure data per CLAUDE.md rule 23)
- **service** — API callers, file handlers, audio processors
- **protocol** — interfaces for swappability
- **view** — SwiftUI views and screens
- **block** — structured content block renderers
- **utility** — parsers, helpers, extensions
- **config** — environment vars, constants, feature flags

## Step 4: Determine the file number

Check the `todo/inventory/` directory for existing files:
- Glob `todo/inventory/*.md`
- Find the highest number N
- New file is `(N+1)-<imagine-slug>.md`
- If no files exist, start at `1-<imagine-slug>.md`
- Derive `imagine-slug` from the user's description (2-3 words, kebab-case, e.g. `ai-chat-service`, `structured-blocks`, `full-stack-chat`)

## Step 5: Write the imagined inventory

Create: `todo/inventory/<N>-<imagine-slug>.md`

The document MUST follow this exact structure:

**Project:** <PROJECT_NAME>
**Created:** <YYYY-MM-DD>
**Generated by:** `/imagine-inventory`
**Vision:** <one-line summary of what user described>

## Instructions

This is a **forward-looking TODO inventory** — none of these items exist yet. It represents everything that needs to be built to deliver the described features.

### How to use this document
- Items are numbered sequentially within each sub-type (restart at 1 per section)
- Status tracks build progress: `todo` → `in-progress` → `done` → `cut` (descoped)
- Run `/imagine-inventory` again to generate additional inventories for new features
- As items get built, update status and add the file path
- Cross-reference with `/get-screen-inventory` to track what's wired on each screen

### Document sections
1. **Summary** — Counts per sub-type with status breakdown
2. **Inventory** — Complete itemised list by sub-type
3. **Dependency Graph** — What blocks what
4. **Build Order** — Recommended implementation sequence
5. **Environment Requirements** — Keys, config, external services needed
6. **StockTake** — Audit trail

### Status definitions
| Status | Meaning |
|--------|---------|
| `todo` | Not started — needs to be built |
| `in-progress` | Currently being worked on |
| `done` | Built and working — file path noted |
| `cut` | Descoped — decided not to build |
| `blocked` | Can't start until dependency is done |

## Rules

- Every item MUST have a planned file path following CLAUDE.md conventions (one type per file, MVVM structure)
- Mark pure data types as needing `nonisolated` per CLAUDE.md rule 23
- Don't duplicate items already tracked in existing screen inventories — reference them
- Every service must have a corresponding protocol for testability/swappability
- Include error types for every service (e.g. `AIServiceError`, `AudioServiceError`)
- Include loading/streaming state types where relevant
- The Build Order must respect the Dependency Graph — nothing starts before its dependencies
- Be thorough but don't pad — every item must serve a real purpose for the described features
- If the user mentions cross-platform, note which items are Swift-only vs shared contract (JSON types)
- Config items should specify WHERE to get the value (API dashboard URL, etc.)
- Use the CURRENT date for all timestamps
