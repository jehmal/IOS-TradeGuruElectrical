# Screen Inventory: $ARGUMENTS

You are a screen inventory agent. Your job is to read every Swift file for a given screen, catalogue every interactive element, and produce a complete inventory showing what has logic, what's empty, and what needs backend before the app gets more complex.

## Step 1: Identify the screen

The user specified: `$ARGUMENTS`

- Set `SCREEN_SLUG` to this value (lowercase, hyphenated)
- Read `rork.json` to get the project name and app path
- Set `PROJECT_NAME` from rork.json (lowercase, hyphenated)
- Find the primary view file: search `ios/**/*View.swift` and `ios/**/*Screen.swift` for a struct name matching the screen slug (e.g. "chat" → `ChatView.swift`)
- If no match, try glob `ios/**/*.swift` and grep for `$ARGUMENTS` case-insensitively

## Step 2: Find ALL related files

From the primary view file:
1. Read it completely
2. Identify every custom type referenced (other views, enums, models, color extensions)
3. Find and read each of those files too
4. Build the complete file list for this screen

## Step 3: Extract every element

Read every file and extract:

### Buttons
For every `Button(action:)` or tappable element:
- Name (what it does)
- SF Symbol icon (if any)
- File + line number
- Current logic: what's inside the `action:` closure. If empty `{}`, mark as "empty"
- Backend needed: YES/NO with reason

### Icons
For every `Image(systemName:)` or `Image()`:
- SF Symbol name or asset name
- File + line
- Purpose (decorative, indicator, button icon)

### Inputs
For every `TextField`, `TextEditor`, `Picker`, `Toggle`, `Slider`, `PhotosPicker`, etc:
- Name and type
- File + line
- What it binds to (`$variable`)
- Current logic: does the value go anywhere on submit?
- Backend needed: YES/NO

### Components (composite views)
For every custom `struct: View` used on this screen:
- Name and file
- Props it takes (`let`, `@Binding`, closures)
- What logic it contains
- Backend needed: YES/NO

### Text Elements
For every `Text()`:
- Content (literal string or dynamic property)
- File + line
- Whether it's dynamic (changes with state)
- Backend needed: only if it should show data from an API

### State Variables
For every `@State`, `@Binding`, `@Observable`, `@Environment`:
- Variable name and type
- Owner file + line
- What it controls

## Step 4: Determine what needs backend/logic

For every element, classify:
- **Has logic** — the action/binding does something real
- **Empty** — the action is `{}` or the value goes nowhere
- **Needs backend** — requires an API call, data model, or persistence

Group these into priorities:
- **Priority 1** — core functionality (can't use the app without it)
- **Priority 2** — essential UX (app works but feels broken without it)
- **Priority 3** — polish (nice to have, not blocking)

## Step 5: Check if inventory file already exists

- If `screens/<SCREEN_SLUG>/inventory.md` already exists:
  - Read it
  - Parse the existing inventory tables to get previously tracked items
  - Merge: keep existing items, update changed ones, mark removed ones as `removed`
  - Update the `## StockTake` section with the new check date
  - Use `Edit` to update (not overwrite)
- If it does not exist:
  - Create the `screens/<SCREEN_SLUG>/` directory if needed
  - Use `Write` to create the file fresh

## Step 6: Write the inventory

Output to: `screens/<SCREEN_SLUG>/inventory.md`

The document MUST follow this exact structure:

```markdown
# <Screen Name> Screen Inventory — <PROJECT_NAME>

**Last updated:** <YYYY-MM-DD>
**Generated by:** `/get-screen-inventory <SCREEN_SLUG>`
**Source files:** <comma-separated list of all Swift files for this screen>

---

## Instructions

This document tracks every interactive element on the **<Screen Name>** screen.

### How to use this document
- Items are numbered sequentially within each category (restart at 1 per section)
- Each item shows: number, name, location (file:line), current logic, and backend status
- Run `/get-screen-inventory <SCREEN_SLUG>` to refresh this inventory
- Use the Priority list at the bottom to plan backend work before adding new screens
- Items removed between scans are marked `removed`, never deleted

### Document sections
1. **Summary** — Quick counts per category with logic/backend breakdown
2. **Buttons** — Every tappable element with its action status
3. **Icons** — Every icon with SF Symbol name and purpose
4. **Inputs** — Every form element with binding and submit status
5. **Components** — Every composite view with props and logic
6. **Text Elements** — Every text with dynamic/static classification
7. **State Variables** — Every @State/@Binding/@Observable with ownership
8. **Action Items** — What needs wiring before adding complexity
9. **Color Tokens** — Design token reference (if color extension exists)
10. **StockTake** — Audit trail of when inventory was checked and what changed

### Status definitions
| Status | Meaning |
|--------|---------|
| `wired` | Has real logic — action does something |
| `empty` | Action closure is `{}` or value goes nowhere |
| `needs-backend` | Requires API call, data model, or persistence |
| `decorative` | Visual only — no logic needed |
| `removed` | Was in previous scan, no longer found |
```

## Rules

- Read the ACTUAL Swift code — don't guess from file names
- Report exact line numbers from the source files
- If a `Button(action: {})` has an empty closure, status is `empty` — not "not implemented"
- Include color tokens only if a color extension file exists for this screen
- Every SF Symbol must use its exact string name (e.g. `line.3.horizontal` not "hamburger icon")
- `wired` means the element actually DOES something (not just exists in code)
- `needs-backend` means it requires an API call, data persistence, or external service
- `decorative` means purely visual — icons, static text, spacers
- Prioritize ruthlessly — Priority 1 means the app literally cannot function without it
- Action Items must include a "Blocked By" column showing dependencies between items
- If appending to an existing file, add a new StockTake row with action `refresh`
- Do NOT remove items from a previous scan — mark them `removed` in status instead
- Use the CURRENT date for all timestamps
- Keep paths relative to project root
