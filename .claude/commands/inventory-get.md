# Inventory Get: $ARGUMENTS

You are an inventory agent. Your job is to scan the entire codebase and produce a complete, numbered inventory of all **$ARGUMENTS** items in this project.

## Step 1: Determine project context

- Read `rork.json` (if it exists) to get the project name. If not found, derive the project name from the working directory name.
- Set `PROJECT_NAME` to the project name (lowercase, hyphenated).
- Set `TYPE` to `$ARGUMENTS` (lowercase, singular form — e.g. "components" → "component").
- The output file is: `inventory/${PROJECT_NAME}-inventory-${TYPE}.md`

## Step 2: Determine search patterns for TYPE

Use this mapping to decide what to scan for. If TYPE is not listed, use the **generic** pattern.

| TYPE | File patterns | Grep patterns | Sub-types |
|------|--------------|---------------|-----------|
| `component` | `**/*.swift`, `**/*.tsx`, `**/*.vue` | `struct \w+: View`, `class \w+View`, `func.*View`, `export.*function\|const.*=.*=>` | view, cell, sheet, modal, control, layout |
| `tool` | `.claude/tools/**`, `**/tools/**`, `**/cli/**`, `**/*.py` | `def \w+\(`, `@mcp.tool`, `sub\.add_parser`, `argparse` | cli, script, utility, mcp-tool |
| `model` | `**/Models/**`, `**/models/**`, `**/*.swift` | `struct \w+:.*Codable`, `class \w+:.*Codable`, `@Observable`, `interface \w+`, `type \w+ =` | data-model, view-model, api-model, enum |
| `service` | `**/Services/**`, `**/services/**`, `**/api/**` | `class \w+Service`, `struct \w+Service`, `protocol \w+`, `actor \w+` | network, storage, auth, analytics |
| `view` | `**/Views/**`, `**/views/**`, `**/screens/**` | `struct \w+View:`, `struct \w+Screen:` | screen, sheet, tab, widget, overlay |
| `viewmodel` | `**/ViewModels/**`, `**/viewmodels/**` | `@Observable.*class`, `class \w+ViewModel` | observable, store, manager |
| `skill` | `.claude/skills/**/SKILL.md` | `^name:`, `^description:` | skill |
| `extension` | `**/Extensions/**`, `**/extensions/**` | `extension \w+`, `protocol \w+` | swift-extension, app-extension |
| `asset` | `**/*.xcassets/**`, `**/assets/**` | `Contents.json` | image, icon, color, data |
| `route` | `**/routes/**`, `**/trpc/**`, `**/api/**` | `\.get\(`, `\.post\(`, `Procedure`, `router` | api-route, page-route, trpc |
| `hook` | `.claude/hooks/**`, `**/hooks/**` | `def \w+`, `hook`, `pre_`, `post_` | pre-hook, post-hook, event |
| `command` | `.claude/commands/**` | `^description:` | slash-command |
| `test` | `**/*Test*`, `**/*test*`, `**/*spec*` | `func test`, `describe\(`, `it\(`, `def test_` | unit, integration, snapshot |
| **generic** | `**/*${TYPE}*`, `**/${TYPE}s/**`, `**/${TYPE}/**` | `${TYPE}`, `class.*${TYPE}`, `struct.*${TYPE}` | (auto-detect from directory names) |

## Step 3: Scan the codebase

1. Use `Glob` to find all files matching the patterns for TYPE.
2. Use `Grep` to search for the grep patterns across matched files.
3. For each match, extract:
   - **Name**: The identifier (class name, struct name, function name, file name)
   - **Path**: Relative file path with line number (e.g. `ios/Views/HomeView.swift:15`)
   - **Sub-type**: Classify into one of the sub-types for this TYPE
   - **Status**: `active` (imported/referenced elsewhere), `unused` (no references found), or `deprecated` (marked deprecated)
   - **Dependencies**: Other inventory items this one imports/references

4. To determine status, for each item do a quick grep for its name across the codebase. If found only in its own file → `unused`. If found in 2+ files → `active`.

## Step 4: Check if inventory file already exists

- If `inventory/${PROJECT_NAME}-inventory-${TYPE}.md` already exists:
  - Read it
  - Parse the existing `## Inventory` section to get previously tracked items
  - Merge: keep existing items, add new ones, mark removed ones as `removed`
  - Update the `## StockTake` section with the new check date
  - Use `Edit` to update (not overwrite)
- If it does not exist:
  - Create the `inventory/` directory if needed
  - Use `Write` to create the file fresh

## Step 5: Write the inventory document

The document MUST follow this exact structure:

```markdown
# ${TYPE} Inventory — ${PROJECT_NAME}

**Last updated:** <YYYY-MM-DD>
**Generated by:** `/inventory-get ${TYPE}`
**Source scope:** <brief description of what was scanned, e.g. "All .swift files in ios/">

---

## Instructions

This document tracks all **${TYPE}** items in the ${PROJECT_NAME} project.

### How to use this document
- Items are numbered sequentially within each sub-type section (restart at 1 per section)
- Each item shows: number, name, path, status, and dependencies
- Run `/inventory-get ${TYPE}` to refresh this inventory
- Manually add items by appending to the appropriate sub-type section
- Items removed between scans are marked `removed`, never deleted

### Document sections
1. **Summary** — Quick counts per sub-type with status breakdown
2. **Inventory** — Complete numbered list segmented by sub-type
3. **Action Items** — Items needing attention (unused, deprecated, or missing)
4. **StockTake** — Audit trail of when inventory was checked and what changed

### Status definitions
| Status | Meaning |
|--------|---------|
| `active` | Referenced in 2+ files — in use |
| `unused` | Only in its own file — may be dead code |
| `deprecated` | Marked deprecated in source |
| `removed` | Was in previous scan, no longer found |
| `pending` | Defined but not yet implemented |

---

## Summary

| Sub-type | Count | Active | Unused | Deprecated | Pending |
|----------|-------|--------|--------|------------|---------|
| ${sub_type_1} | N | N | N | N | N |
| ${sub_type_2} | N | N | N | N | N |
| **Total** | **N** | **N** | **N** | **N** | **N** |

---

## Inventory

### ${Sub-type 1}

| # | Name | Path | Status | Dependencies |
|---|------|------|--------|--------------|
| 1 | ExampleName | `path/to/file.swift:42` | active | OtherItem, AnotherItem |

### ${Sub-type 2}

| # | Name | Path | Status | Dependencies |
|---|------|------|--------|--------------|
| 1 | ... | ... | ... | ... |

---

## Action Items

Items needing attention from this scan:

| # | Item | Status | Action Needed |
|---|------|--------|---------------|
| 1 | <name> | unused | Verify if needed or remove |
| 2 | <name> | deprecated | Plan migration/removal |
| 3 | <name> | pending | Implement |

---

## StockTake

| Date | Action | Added | Removed | Total | Notes |
|------|--------|-------|---------|-------|-------|
| YYYY-MM-DD | initial scan | N | 0 | N | |
```

## Rules

- Number items sequentially WITHIN each sub-type (restart at 1 for each section)
- Use the CURRENT date for all timestamps
- If appending to an existing file, add a new row to StockTake with action `refresh`
- Do NOT remove items from a previous inventory — mark them `removed` in status instead
- Keep paths relative to project root
- If a type has no sub-types detected, use a single "general" sub-type
- Be thorough — scan the ENTIRE project, not just obvious directories
- For Swift projects, pay attention to MVVM structure (Views, ViewModels, Models, Services)
- The Action Items section should only list items that need human attention — not every active item
