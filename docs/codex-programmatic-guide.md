# Running Codex Programmatically

Codex supports automation in two practical ways: non-interactive batch runs with `codex exec`, and tool-style integration with `codex mcp-server`. Use `exec` when a human, script, CI job, or another agent wants to send instructions and receive an answer. Use `mcp-server` when you already have an MCP-capable orchestrator and want Codex exposed over stdio.

The simplest pattern is a one-shot command:

```bash
codex exec "Summarize this repository and list the top three risks."
```

You can also pipe instructions through standard input, which is usually better for long prompts generated by scripts or agents:

```bash
printf '%s\n' 'Review the repo and propose a release checklist.' | codex exec -
```

For reliable automation, set the working directory explicitly so Codex operates on the intended repository:

```bash
codex exec -C /path/to/repo "Run tests and explain failures."
```

If you need additional writable directories, add them with `--add-dir`. If you are outside a Git repository, `--skip-git-repo-check` allows the run. For temporary jobs that should not persist session files, add `--ephemeral`.

Structured output matters when another program will read the result. `--json` streams events as JSONL to stdout, which is the best option for agent frameworks and log pipelines. `--output-last-message` writes only the final user-facing answer to a file. `--output-schema` lets you provide a JSON Schema describing the exact shape of the final response, which is the cleanest way to make downstream parsing predictable.

Example:

```bash
codex exec \
  --json \
  --output-schema ./schema.json \
  --output-last-message ./result.json \
  "Return a JSON deployment plan for this project."
```

For humans automating from shell scripts, a good default is sandboxed execution with an explicit directory and captured output. For AI agents, prefer stdin for prompts, `--json` for event streaming, and `--output-schema` when the result will feed another tool.

Permissions and sandboxing are the main operational choice. If you want unattended execution without approval prompts but still want a sandbox, use the global approval flag before `exec`:

```bash
codex -a never exec --sandbox workspace-write "Apply the requested fix and run safe checks."
```

That keeps Codex inside the workspace sandbox while preventing interactive approval pauses. If you want the convenience preset for sandboxed automation, `codex exec --full-auto ...` uses `workspace-write`, but it is not the same as fully disabling approvals.

If you want to bypass both approvals and sandboxing entirely, Codex exposes a dedicated flag:

```bash
codex exec --dangerously-bypass-approvals-and-sandbox "Perform the task."
```

Use that only inside an external safety boundary such as a disposable container, locked-down VM, or tightly scoped CI runner. It gives the agent unrestricted command execution.

For deeper agent-to-agent integration, start Codex as an MCP server:

```bash
codex mcp-server
```

This runs over stdio, so another MCP client can launch Codex as a subprocess and communicate through the protocol instead of screen-scraping terminal output.

In practice, the safest default for programmatic use is: `codex -a never exec --sandbox workspace-write`, plus `--json` and `--output-schema` when another system needs deterministic output. Reach for full bypass only when you have already sandboxed the environment outside Codex itself.
